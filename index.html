Data is stored on SANBI's disk array (next generation pictured here) organised into a project folder in shared disk storage. By the end of the project this shared workspace contained 632 GB of data.
Where did all this extra data come from? 1. Contributed from publicly available data sources and 2. generated from analysis that split, recombine, copy and transform incoming data.

An idealised bioinformatics data pipeline

Ideally all computational research is reproducible. We should have a turnkey solution where we input initial data, hit go, and let the computers do the work. Scientific workflow management systems promise to enable this.

A description of the real world data pipeline

Unfortunately available scientific workflow management systems currently lack the power, control and efficiency necessary for our data analysis. The reality is a system of scripts that embody stages of the pipeline, manually executed, with their execution tracked in READMEs (that you hope are comprehensive). In addition, data analysis tends to move between clusters - SANBI, CHPC and UWC Astrophysics.

Analysis source code is easily tracked using version control

Workflow scripts (and READMEs) are synchronised with version control system (git/bitbucket.org) that tracks edits (versioning) and provenance of changes.

Data is largely managed as aggregate "disk usage"

How to manage all this data? Current data management is focused on aggregate of disk space (metric is availability). To be honest no one outside the project knows what the different data collections comprise of. Within the project disk usage is (at best) documented in READMEs or (at worst) ephemeral email exchanges. Project storage aggregates "1 week storage" (data immediately in use for analysis) and "1 year storage" (data used to feed into the analysis goals of the project as a whole), the distinction is not managed.

Results extracted from flat files populate a relational database

As data accumulates, results are extracted from flat files and structured into a relational database.

A genome browser helps visualise predicted genes

And "genome features" visualised in a genome browser. Visualisation forms a key part of data analysis and quality control.

The Bass Explorer web portal

And finally the output of the project is published using a web portal. Raw data outputs are also published to the NCBI Genbank repository. What to do with data that is not published to Genbank? Historically data has been published on SANBI FTP site, but this is not very findable and not very durable. For data supporting paper, Datadryad is an option, also Figshare: data publication is a new field for SANBI.
